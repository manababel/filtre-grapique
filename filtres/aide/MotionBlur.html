Échantillonnage Vectoriel

Documentation - Algorithme MotionBlur

Simulation physique de vitesse d'obturation par intégration rectiligne orientée

1. Pourquoi utiliser cette implémentation de Motion Blur ?

  • Projection Trigonométrique Précise : Le filtre convertit l'angle utilisateur (0 à 360 degrés) en radians pour générer un vecteur de direction unitaire (dx, dy) à l'aide des fonctions Cos() et Sin(). Cela permet de balayer l'image dans n'importe quelle direction spatiale avec une précision absolue.
  • Algorithme non séparable optimisé : Contrairement à un flou gaussien, un flou directionnel incliné ne peut pas être séparé simplement en deux passes orthogonales (X puis Y). Votre code résout le problème en effectuant un unique voyage rectiligne centré sur le pixel cible, minimisant la structure algorithmique.
  • Optimisations bas niveau (Bit Shift) : Pour maximiser la vitesse, le nombre total d'échantillons (la taille de la ligne de flou) est calculé via l'opérateur de décalage de bits gauche radius << 1 au lieu d'une multiplication standard par 2.

2. Les Inconvénients et Limites du Filtre

  • Effet de crénelage (Aliasing) sur les coordonnées entières : Comme le vecteur de déplacement (dx, dy) utilise des valeurs flottantes décimales mais que la lecture finale en mémoire est convertie en entiers via Int(xi), l'échantillonnage subit de légères approximations. Sur de grands rayons, cela peut créer des saccades visuelles ou un effet d'escalier sur le flou, faute d'une véritable interpolation bilinéaire.
  • Surcharge algorithmique redondante — O(N * (2R + 1)) : Pour chaque pixel de l'image, le moteur doit recalculer l'intégralité du segment de droite. Aucun tampon intermédiaire ne sauvegarde les calculs des pixels adjacents. Le coût machine explose de manière strictement linéaire lorsque le rayon augmente.
  • Distorsion des bords par "Clamping" directionnel : Lorsque la ligne de balayage sort du cadre de l'image, le système fige les coordonnées sur les pixels des bordures. Dans le cas d'un flou directionnel, cela produit un effet d'étirement "baveux" très prononcé et asymétrique sur les limites de la texture.

Le Noyau Vectoriel : Précalcul des Composantes

Cette section prépare les facteurs de déplacement du repère pour transformer la boucle interne en un simple suivi de trajectoire.

; Conversion de l'angle en radians et calcul de la taille du segment :
Protected angle.f = \option[1] * #PI / 180.0
Protected dx.f = Cos(angle)
Protected dy.f = Sin(angle)

Protected size = (radius << 1) + 1  ; Équivalent à : (radius * 2) + 1
Protected coeff.f = 1.0 / size       ; Coefficient multiplicateur pour la moyenne

4. Le Filtre : Comment ça marche ?

L'algorithme n'applique pas une matrice carrée. Il trace une ligne droite imaginaire à travers le pixel et fait la moyenne de tous les points rencontrés le long de ce segment.

Calcul du point de départ du segment

Pour que le flou soit parfaitement équilibré autour du pixel d'origine, le code recule d'abord la position de lecture initiale en multipliant le rayon par les pas directionnels :
xi = x - dx * radius et yi = y - dy * radius. Ensuite, la boucle avance pas à pas vers l'avant en ajoutant dx et dy à chaque échantillon.

Calcul des incréments successifs :

Pas_X = Cos(Angle)  |  Pas_Y = Sin(Angle)

Position(k) = Point_Départ + k * (Pas_X, Pas_Y)

À chaque itération de la boucle k, la coordonnée glisse le long du vecteur. Les valeurs de Rouge, Vert et Bleu sont extraites de l'image source, cumulées, puis multipliées par le coefficient inverse (coeff) pour générer l'effet de vitesse.

Le Clamping de protection d'index

Pour éviter les débordements de mémoire lors de la lecture des pixels en dehors de l'écran, le code applique une structure conditionnelle stricte :

If xi < 0 : xiClamped = 0 : ElseIf xi >= w : xiClamped = wMinus1 : Else : xiClamped = Int(xi) : EndIf

Cette sécurité tronque la valeur décimale en position entière valide (Int(xi)) tout en s'assurant que le résultat reste verrouillé entre l'index 0 et la largeur maximale de l'image (wMinus1).

Logique Interne de l'Algorithme (Pseudo-code Synchrone)

Voici le cheminement logique du processus d'intégration rectiligne :

Entrée: Image Source, Image Cible, Rayon, Angle en Degrés

// 1. PREPARATION DES VECTEURS
Angle_Radians = Angle * PI / 180.0
Pas_X = Cos(Angle_Radians)
Pas_Y = Sin(Angle_Radians)
Nombre_Echantillons = (Rayon * 2) + 1
Coefficient_Moyenne = 1.0 / Nombre_Echantillons

// 2. BALAYAGE DE LA MATRICE
Pour chaque Ligne Y de 0 à Hauteur - 1
    Pour chaque Colonne X de 0 à Largeur - 1
        
        Somme_R = 0 : Somme_G = 0 : Somme_B = 0
        
        // Reculer au point de départ de la trajectoire
        Curseur_X = X - (Pas_X * Rayon)
        Curseur_Y = Y - (Pas_Y * Rayon)
        
        // Échantillonner le long de la ligne droite
        Pour k = 0 To Nombre_Echantillons - 1
            
            // Encapsulation de sécurité (Clamping)
            X_Valide = Verrouiller_Bornes(Int(Curseur_X), 0, Largeur - 1)
            Y_Valide = Verrouiller_Bornes(Int(Curseur_Y), 0, Hauteur - 1)
            
            // Accumulation des couleurs
            Pixel_Couleur = Lire_Pixel(Image_Source, X_Valide, Y_Valide)
            Somme_R + Extraction_Rouge(Pixel_Couleur)
            Somme_G + Extraction_Vert(Pixel_Couleur)
            Somme_B + Extraction_Bleu(Pixel_Couleur)
            
            // Avancer vers le point suivant sur le vecteur
            Curseur_X + Pas_X
            Curseur_Y + Pas_Y
        Next k
        
        // Calcul de la moyenne et réassemblage
        R_Final = Somme_R * Coefficient_Moyenne
        G_Final = Somme_G * Coefficient_Moyenne
        B_Final = Somme_B * Coefficient_Moyenne
        
        Écrire_Pixel(Image_Cible, X, Y, Réassembler_ARGB(255, R_Final, G_Final, B_Final))
        
    Next X
