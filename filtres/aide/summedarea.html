Documentation - Algorithme Summed-Area Table

Génération et application d'une Image Intégrale pour filtres en temps réel

1. Pourquoi utiliser une Summed-Area Table (SAT) ?

  • Complexité Algorithmique Temporelle en O(1) : C'est la force absolue de cet algorithme. Contrairement à une boîte de flou classique dont le temps de calcul explose lorsque le rayon augmente, la méthode SAT traite un pixel en un temps strictement constant. Que le rayon soit de 2 pixels ou de 500 pixels, la vitesse d'exécution reste identique.
  • Fondation pour les filtres complexes : Au-delà du flou de boîte ultra-rapide, l'image intégrale permet de calculer instantanément des variances locales, des moyennes glissantes, ou d'alimenter des algorithmes de vision par ordinateur comme la détection de visages (méthode de Viola et Jones).

2. Limites de la méthode SAT & Alternatives

Inconvénients majeurs de l'algorithme SAT :
  • Consommation mémoire massive (RAM) : La SAT nécessite d'allouer quatre buffers de travail temporaires de la taille exacte de l'image (un par canal de couleur). Pour une image de 4K ($3840 \times 2160$), cela représente une surconsommation de plus de 132 Mo de RAM uniquement pour stocker les données de calcul intermédiaires.
  • Risque de dépassement de capacité (Integer Overflow) : Sur de très grandes images, l'addition cumulée continue des valeurs de pixels (0-255) peut dépasser la limite maximale d'un entier 32 bits standard. L'utilisation de types flottants ou d'entiers 64 bits sécurise le calcul mais alourdit la bande passante mémoire.
  • Esthétique "angulaire" de faible qualité : Le filtre SAT produit exclusivement un Flou Box (flou de boîte carré). Graphiquement, les contours floutés laissent apparaître des formes cubiques inesthétiques par rapport à la douceur d'un flou Gaussien.
Comparatif des alternatives technologiques

Si la méthode SAT offre un temps d'exécution stable, d'autres approches permettent d'obtenir un meilleur rendu visuel ou une exécution plus fluide selon vos besoins graphiques :

Pour une qualité graphique supérieure : Le Flou Gaussien Séparé

Le flou Gaussien applique une pondération en cloche : les pixels proches du centre ont plus d'importance que les pixels éloignés. Le rendu visuel est beaucoup plus naturel et soyeux, sans l'effet "bloc" du flou Box.

  • L'astuce mathématique : Bien qu'un flou Gaussien 2D soit très lourd, il est séparable. On peut appliquer un premier flou de noyau 1D horizontal, puis réappliquer ce même noyau 1D verticalement sur le résultat. La complexité brute chute de $O(N^2)$ à $O(2N)$.
  • Quand l'utiliser : Pour le floutage d'arrière-plans d'interfaces utilisateur (UI), les rendus de profondeur de champ cinématiques ou les lissages photographiques de haute précision.

Pour une vitesse maximale sur petits rayons : Le Box Blur Glissant (Sliding Window)

Si vous souhaitez conserver le rendu d'un Flou Box mais que le rayon utilisé dépasse rarement 15 à 20 pixels, le Sliding Window Box Blur est une alternative redoutable à la SAT.

  • Le principe : Au lieu d'allouer des buffers géants pour toute l'image, on déplace une fenêtre d'analyse le long de la ligne. À chaque déplacement d'un pixel vers la droite, on soustrait la valeur du pixel qui sort de la fenêtre à gauche et on ajoute celle du pixel qui entre à droite.
  • Les avantages clés :
    • Zéro surcoût mémoire : Pas d'allocations massives de buffers SAT, la consommation RAM reste minimale.
    • Vitesse brute : Parfaitement adapté aux instructions vectorielles (comme le SSE2 / AVX2), il s'exécute souvent plus rapidement que la SAT sur de petites fenêtres car il ne nécessite pas les deux passes d'initialisation (horizontale et verticale) requises pour fabriquer l'image intégrale.

3. Exemples d'implémentation (PureBasic)

Option A : Utilisation Directe (Fonction Standard)

Idéal pour appliquer le flou de zone directement sur une image à l'aide d'un rayon prédéfini.

IncludeFile "filtres.pbi"
UseModule filtres

If OpenWindow(0, 0, 0, 800, 600, "Exemple SAT...", #PB_Window_SystemMenu | #PB_Window_ScreenCentered)
  
  SourceImage = 1
  CibleImage  = 2
  AucunMasque = 0
  Rayon       = 12 ; Le rayon du flou de boîte
 
  file$ = OpenFileRequester("Choisir une Image","","",0)
  If load_image_32(SourceImage , file$) = 0
    End
  Else
    ResizeImage(SourceImage, 800, 600)
  EndIf
  
  CopyImage(SourceImage, CibleImage)

  ; Appel de la procédure standard simplifiée
  SummedArea(SourceImage, CibleImage, AucunMasque, Rayon)
  
  StartDrawing(WindowOutput(0))
    DrawImage(ImageID(CibleImage) ,  0 , 0)
  StopDrawing()
  
  Repeat
    Event = WaitWindowEvent()
  Until Event = #PB_Event_CloseWindow
EndIf

Option B : Utilisation Avancée (Via Contexte de Filtre / SummedAreaEx)

Recommandé si vous devez intégrer ce filtre dans une chaîne de traitement ou automatiser la gestion des canaux de couleur via la structure globale.

IncludeFile "filtres.pbi"
UseModule filtres

; Préparation du contexte du filtre
Clear_Data_Filter(FilterCtx)
FilterCtx\option[0] = 25 ; Définition du rayon d'action dans le contexte

Set_Source(SourceImage)
Set_Cible(CibleImage)
Set_Mask(AucunMasque)

; Execution de l'algorithme complet (Allocation, SAT_Create, SAT_Apply, Free)
SummedAreaEx(FilterCtx.FilterParams)

4. Le Filtre : Comment ça marche ?

L'algorithme de Summed-Area Table fonctionne en deux passes mathématiques distinctes : d'abord la génération de tables cumulatives (Image Intégrale), puis la reconstruction géométrique de la zone pour calculer la moyenne.

1. La Phase de Création (Sommes Cumulées)

On crée quatre tables de données distinctes (une pour chaque canal de couleur : Alpha, Rouge, Vert, Bleu). Chaque case (X, Y) de ces tables ne contient pas la couleur du pixel d'origine, mais la somme totale de tous les pixels situés au-dessus et à gauche de cette case.

2. L'astuce géométrique d'extraction (O(1))

Pour trouver la somme des pixels d'un rectangle virtuel — défini par le rayon du flou — l'ordinateur n'a plus besoin de lire tous les pixels à l'intérieur. Il lui suffit de lire seulement 4 valeurs clés dans la table SAT grâce à la macro GetRectSum.

Les 4 points clés SAT
AB
CD

Formule mathématique de résolution :

Somme_Zone = D - B - C + A

En prenant la valeur cumulée au coin inférieur droit (D), on retire les rectangles excédentaires du haut (B) et de gauche (C). Le rectangle supérieur gauche (A) ayant été soustrait deux fois, on le ré-additionne pour équilibrer le calcul.

Logique interne de l'algorithme (Pseudo-code)

Voici le comportement exact transcrit depuis votre filtre PureBasic :

// --- PASSE 1 : CRÉATION DE LA IMAGE INTÉGRALE (SAT) ---
Pour chaque ligne Y de l'image d'origine
    Pour chaque colonne X (de 1 à Largeur - 1)
        // Cumul horizontal par ligne
        SAT_Canal[X, Y] = Pixel_Source_Canal + SAT_Canal[X - 1, Y]
    Next
Fin Pour

Pour chaque ligne Y (de 1 à Hauteur - 1)
    Pour chaque colonne X (de 0 à Largeur - 1)
        // Cumul vertical cumulatif
        SAT_Canal[X, Y] = SAT_Canal[X, Y] + SAT_Canal[X, Y - 1]
    Next
Fin Pour


// --- PASSE 2 : APPLICATION ET LECTURE ---
Pour chaque pixel (X, Y) de l'image finale
    // Délimiter les limites de la boîte de voisinage
    x1 = X - Rayon - 1 : x2 = X + Rayon
    y1 = Y - Rayon - 1 : y2 = Y + Rayon
    
    // Bornage de sécurité aux limites de la texture
    Ajuster_Limites_Bords(x1, y1, x2, y2)
    
    // Nombre total de pixels englobés dans le rectangle courant
    Nombre_Pixels = (x2 - x1) * (y2 - y1)
    
    // Extraction instantanée de la somme par canal via la macro (D - B - C + A)
    SommeA = Obtenir_Somme_SAT(SAT_Alpha, x1, y1, x2, y2)
    SommeR = Obtenir_Somme_SAT(SAT_Rouge, x1, y1, x2, y2)
    SommeG = Obtenir_Somme_SAT(SAT_Vert,  x1, y1, x2, y2)
    SommeB = Obtenir_Somme_SAT(SAT_Bleu,  x1, y1, x2, y2)
    
    // Ecriture du pixel reconstruit avec sa moyenne locale
    Pixel_Destination = ARGB(SommeA / Nombre_Pixels, SommeR / Nombre_Pixels, ...)
